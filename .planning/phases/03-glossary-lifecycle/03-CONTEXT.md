# Phase 3: Glossary Lifecycle - Context

**Gathered:** 2026-06-01
**Status:** Ready for planning

<domain>
## Phase Boundary

Create a reviewable initial glossary and evolve it safely as translation progresses. This includes generating an initial glossary automatically from crawled chapters via LLM/agent scan of the first few chapters, merging progressive new term proposals from sequential translation workers, reporting duplicate and conflicting proposals in a deterministic report, and enabling safe manual edits and snapshots of the glossary. This phase does not cover downstream chapter translation, translation QA, or format export.

</domain>

<decisions>
## Implementation Decisions

### Glossary Storage Format and Schema
- **D-01:** Implement a Structured Term Schema (Option B) for `glossary.yaml` stored in the book workspace root (e.g., `books/<book-slug>/glossary.yaml`). Each term maps to a structured dictionary containing:
  - `translation`: str (Vietnamese mapping)
  - `category`: str (one of `character`, `sect`, `location`, `item`, `cultivation`, `other`)
  - `source`: str (source identifier: `manual`, `initial_generation`, or `chapter_N_proposal`)
  - `is_canonical`: bool (lock flag; defaults to `false` for auto-generated/proposed terms, and set to `true` for manual edits/additions)
  - `note`: str | None (optional description or context)

### Initial Glossary Generation Strategy
- **D-02:** Implement Option B for initial glossary generation. Automatically scan the first few raw chapter files (e.g., Chapters 1 to 3) using an LLM or agent-native worker to extract and translate core names, places, and terms in their literary context, storing them as the initial `glossary.yaml` file with `is_canonical: false` and `source: initial_generation`.

### Progressive Merging and Conflict Handling
- **D-03:** Implement Option A for progressive merging of chapter proposals. It is a non-blocking merge:
  - If a worker proposes a term that already exists in the glossary:
    - If the proposed translation differs, preserve the existing translation (prioritizing `is_canonical: true`).
    - Write a detailed conflict record to a dedicated `reports/glossary-conflicts.yaml` containing the term, existing translation/source, proposed translation/source, and chapter number.
    - Print a warning message to the console, but do NOT stop or fail the active translation run.

### Manual Edits and Snapshotting
- **D-04:** Implement Option A for manual edits and snapshots.
  - Automatically create backup snapshots of the `glossary.yaml` file under `books/<book-slug>/checkpoints/glossary-snapshots/chapter-N.yaml` before merging any new chapter-level proposals.
  - Manually edited or user-added terms in `glossary.yaml` are treated as canonical by setting `is_canonical: true` to prevent any automated proposal from overwriting them.

### the agent's Discretion
No implementation decisions were delegated during discussion.

</decisions>
