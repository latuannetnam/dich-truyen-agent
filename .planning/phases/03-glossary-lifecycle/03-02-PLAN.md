  <behavior>
    - Merging proposals creates a snapshot of the current `glossary.yaml` in `checkpoints/glossary-snapshots/chapter-{chapter_id:04d}.yaml`.
    - If a proposed term does not exist, it is added with `is_canonical=False` and `source="chapter_N_proposal"`.
    - If a term exists and translations match, metadata is updated but no conflict is logged.
    - If a term exists and translations differ:
      - The existing translation is strictly preserved (especially if `is_canonical=True`).
      - A conflict is logged in `reports/glossary-conflicts.yaml`.
      - A warning is printed to the console, and the run continues without blocking.
  </behavior>
  <action>
    Implement `merge_glossary_proposals(workspace_root: Path, chapter_id: int, proposals: dict[str, GlossaryTerm]) -> OperationResult` in `glossary.py`.
    Load `BookGlossary` from `glossary.yaml` (fallback to empty if missing).
    Write the snapshot backup file first using `atomic_write_yaml`.
    Process the merges using the duplicate and conflict detection logic:
    - Track existing term lists.
    - Read/write `reports/glossary-conflicts.yaml` through `GlossaryConflictReport`.
    - Print warning messages for every new mismatch.
    - Write updated glossary and conflicts atomically.
    Write unit tests in `tests/test_glossary.py` to cover progressive merges, matching entries, mismatching conflicts, console warnings, and snapshot existence.
  </action>

  </verify>
