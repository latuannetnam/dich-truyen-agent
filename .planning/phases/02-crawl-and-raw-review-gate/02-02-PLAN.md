  <behavior>
    - Timeout, connect, and `5xx` failures retry at most three attempts with delays derived from configurable exponential backoff.
    - A successful chapter atomically writes UTF-8 raw text, records SHA-256 in state, then invokes the fixed chapter sleeper with default `3`.
    - A chapter still failing after retries records an error and stops the loop before any later chapter fetch.
    - Resume retries the failed chapter and skips already hash-valid completed chapters.
  </behavior>
  <action>
    Implement a narrow retry classifier and retry loop around HTTP chapter fetching in
    `crawl_batch.py`. Count total attempts, not retries: default maximum is `3`. Retry timeout,
    connection, and `5xx` failures with `backoff_seconds * 2 ** (attempt - 1)`. Do not retry
    non-recoverable `4xx`.

    After validated extraction, call `atomic_write_text()` for the canonical raw file, compute
    SHA-256, atomically persist the completed raw `StageRecord`, and invoke injected
    `sleep(settings.chapter_delay_seconds)`. On exhausted failure, atomically persist
    `StageStatus.ERROR`, diagnostic reason, and current chapter metadata; stop without fetching
    later chapters. Use injected transport and sleep spies so offline tests never wait.
  </action>

    <automated>uv run pytest tests/test_crawl_batch.py -q -k "retry or delay or stop or resume"</automated>
  </verify>
  <acceptance_criteria>

    - Chapter pacing spy records `3` seconds by default after each successful chapter only.
    - A persistent chapter-2 failure proves chapter 3 is never fetched.
    - After repair, resume skips chapter 1 and retries chapter 2.
    - `uv run pytest tests/test_crawl_batch.py -q -k "retry or delay or stop or resume"` exits `0`.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| External response -> retry/fallback decision | Statuses and page shapes must not trigger unsafe endless retries or broad browser escalation. |
| Extracted chapter -> canonical raw file | Unvalidated anti-bot or empty content must never be promoted. |
| Mutable state -> resume position | Resume must validate hashes and preserve ordering before skipping work. |
| Diagnostic HTML -> agent context | Full external HTML stays on disk and only compact paths return to the agent. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-04 | Tampering | `crawl_batch.py` resume | mitigate | Reuse strict Phase 1 resume hash validation and stop at the first unresolved chapter gap. |
| T-02-05 | Denial of Service | retry and browser fallback | mitigate | Bound attempts to three, use configurable backoff, and permit Playwright only for validated rendered-content failure. |
| T-02-06 | Information Disclosure | diagnostics and `OperationResult` | mitigate | Persist HTML diagnostics on disk; expose only compact reason, progress, URL, and report paths. |
| T-02-07 | Tampering | CAPTCHA/auth pages | mitigate | Detect failed expected extraction plus blocking response/page shape before raw promotion; test valid Cloudflare-residue non-blocking case. |
</threat_model>
